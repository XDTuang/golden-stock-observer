# 腾讯云托管「兜金观测」盘后自动更新方案

> 目标：把每日盘后数据更新（fetch_pool → 金钻扫描 → 门控 → 精简 → 发布 GitHub Pages + CloudBase）**整体搬到腾讯云自动跑**，本地 Mac 退居「监听 / 兜底」角色，不再承担更新主链路。

---

## 一、现状与关键约束（来自实测）

| 项 | 现状 | 对云端托管的影响 |
|---|---|---|
| 全链路耗时 | ~15 分钟；`gate_scan.py` 曾因网络卡 **21 分钟** | 云函数免费版 3s 超时**不可能**；标准版 900s 也偏紧，需给 `gate_scan` 加硬超时 |
| 数据源 | 腾讯 `gtimg`（腾讯自家）+ 同花顺 `thsdk`（公网 HTTP） | 腾讯云网络访问 `gtimg` 比本地更稳；`thsdk` 走公网，云上同样可达 |
| 产物发布 | GitHub Pages（需 `XDTuang` SSH key）+ CloudBase Hosting（`tcb` CLI） | 云机需持有 GitHub SSH 私钥 + `tcb` 登录态 |
| 源码管理 | **所有 `*.py` 被 `.gitignore` 忽略**（仅 `fetch_pool.py` 强跟踪） | `git clone` 拿不到流水线 `.py`，需单独把脚本同步到云机 |
| 闸门保护 | `update_data.sh` 内置 `should_update_now()`（非交易日/盘中自动跳过） | 云端 cron 即使周末触发也会安全跳过，无需人工判断 |

---

## 二、推荐方案：腾讯云轻量应用服务器（Lighthouse）+ cron

把你本地那套 `local_update.sh` 原样搬到一台云服务器上，由系统 cron 每日触发。**这是成本最低、最稳、最贴近现状的方案**——本质是「本地机搬上云」。

### 实施步骤
1. **开实例**：腾讯云控制台 → 轻量应用服务器 → 新建（Ubuntu 22.04，2 核 2G 起；首单常有 ¥40–60/月档）。
2. **装运行环境**：
   ```bash
   sudo apt update && sudo apt install -y python3.11 python3.11-venv git rsync
   npm i -g @cloudbase/cli        # tcb CLI（发布 CloudBase 用）
   ```
3. **源码上云**（关键：`.py` 被 gitignore，必须带脚本一起）：
   - 最简单：从本地 Mac 一次性 `rsync -az --exclude='.git' /Users/samt/golden_stock_observer/ 云机:/opt/golden_stock_observer/`
   - 或建一个**私有** GitHub 仓库专门收纳含 `.py` 的完整工程，`git clone` 到云机（推荐长期维护）
4. **Python 依赖**：在云机建 venv 并 `pip install` `thsdk` 等（同本地 `requirements`）。
5. **凭证**：
   - GitHub：`ssh-keygen` 生成密钥，公钥加入 `XDTuang` 账号；`git remote` 指向 `git@github.com:XDTuang/golden-stock-observer.git`
   - CloudBase：把本地 `~/.tcbcli.data`（或等效登录态）拷到云机，或在云机跑一次 `tcb login`；`deploy_cloudbase.sh` 读环境变量 `CLOUDBASE_ENV_ID`
6. **cron（工作日 17:35，留足收盘后数据落地时间）**：
   ```cron
   35 17 * * 1-5  cd /opt/golden_stock_observer && bash local_update.sh >> /var/log/goldenstock_update.log 2>&1
   ```
7. **先手动跑一次**验证全链路，再放开 cron。

## 三、备选方案：CloudBase 云函数（标准版 900s）+ 定时触发器

- **适用**：不想管服务器、接受一定脆弱性。
- **做法**：把流水线打成云函数（依赖打包进层），用**定时触发器**每天 17:35 触发。
- **硬伤**：
  - 免费版 3s 超时，直接否决；标准版 900s 仍可能扛不住 15min 主链路 + `gate_scan` 抖动。
  - 需在函数里打包 `thsdk` 依赖、注入 GitHub 凭证与 `tcb` 登录态，配置繁琐。
  - 函数内 `git push` GitHub Pages 需要持久化 SSH key。
- **结论**：仅当 Lighthouse 成本都不愿出时考虑，且务必把 `gate_scan`/`update_data` 拆成可中断续跑的子函数。**不作为首选。**

## 四、本地 Mac 的角色：监听 + 兜底

本地不再跑 `local_update.sh` 主链路，改为轻量「哨兵」：

1. **监听脚本**（launchd 每日 18:10 跑）：轮询 CloudBase / GitHub Pages 的 `data_date`，若：
   - 站点不可达，或
   - `data_date` 落后于「最新交易日」超过 1 天
   → 发通知（微信/邮件/系统通知）。
2. **兜底**（可选，防御性）：监听脚本发现云机失效时，自动本地补跑一次 `local_update.sh`，保证数据不中断。

这样**本地完全不依赖**，只在云端异常时才出手。

---

## 五、上线前必须做的 3 个改造

1. **`.py` 源码上云**：`rsync` 或私有仓库，确保云机有完整流水线脚本（当前 `.gitignore` 会漏掉它们）。
2. **给 `gate_scan` 加硬超时**（防无人值守时卡死整条链路）：
   在 `update_data.sh` 的 Step 2.6 改为：
   ```bash
   timeout 600 "$PYTHON" gate_scan.py --daily || echo "  ⚠️ 门控扫描超时(600s)，跳过"
   ```
3. **凭证落盘**：云机预置 GitHub SSH 私钥 + `tcb` 登录态；`CLOUDBASE_ENV_ID` 写入 cron 环境或 `local_update.sh`。
